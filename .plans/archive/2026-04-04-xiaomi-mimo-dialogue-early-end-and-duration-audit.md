## Goal

Determine whether the Xiaomi MiMo dialogue output is ending early or reporting the wrong audio length because of our config/prompt/validator contract, or because of upstream model grounding and timing behavior.

---

## Overview

The whole-asset dialogue finalization honesty lane is now complete: the pipeline no longer inflates partial Xiaomi output into falsely complete metadata. That means the remaining problem can be investigated cleanly. The live artifact is now honestly showing that Xiaomi reported a shorter `totalDuration` than the real extracted audio runtime, so the next step is to inspect why that happens.

This tranche should focus on the request-generation side and the model-facing contract. We want to inspect the exact Xiaomi/OpenRouter dialogue artifact, the raw provider capture, the prompt expansion generated by the local validator/tool loop, the active config values, and the source path that assembles the request. The core question is whether our own settings are nudging the model to stop early or misstate duration, or whether the provider/model is simply weak at long-form whole-asset audio grounding and time estimation.

The output of this tranche should be a source-backed diagnosis and a ranked shortlist of the smallest next implementation experiments. We should avoid broad speculative prompt changes until we know which seam is actually most suspicious.

- **The whole-asset prompt does not tell the model the real file length.** `buildTranscriptionPrompt()` asks for timestamps and includes a short schema example whose `totalDuration` is `30.5`, but it never injects the real measured audio runtime or any explicit instruction like “the attached audio is 140.042449s long; do not estimate shorter” (`server/scripts/get-context/get-dialogue.cjs`, captured prompt `_meta/ai/_prompts/d5e8ebc829bffa0a9b0b5d68decc5aedaf436fb78689372784cba8eb21a8bae4.json`). That makes wrong duration estimation plausible, especially for a model already showing weak long-form audio grounding.
- **The local validator/tool loop does not enforce duration honesty or full-tail coverage.** The acceptance contract only requires schema-valid JSON after `validate_dialogue_transcription_json` returns `valid=true`; it does not compare `totalDuration` to ffprobe duration, require the last segment/coverage to reach the asset end, or reject transcripts that stop early but remain structurally valid (`server/lib/local-validator-tool-loop.cjs`, `server/lib/phase1-validator-tools.cjs`, `server/lib/structured-output.cjs`). A transcript that ends at `121s` with `totalDuration: 121` is still validator-valid.
- **The tool loop is unlikely to be the cause of shortening.** Whole-asset dialogue defaults to `maxTurns: 4` and `maxValidatorCalls: 3`, but the audited run succeeded on the first provider turn and first validator call. The loop also auto-validates final JSON candidates and returns them when the normalized value matches; it does not trim content on success (`server/scripts/get-context/get-dialogue.cjs`, `server/lib/local-validator-tool-loop.cjs`, raw capture `phase1-gather-context/raw/ai/dialogue-transcription/attempt-01/capture.json`).
- **Token/timeout settings are generous enough that an obvious local cap is unlikely.** The request used OpenRouter `max_tokens: 25000`, translated from adapter params, plus `reasoning.effort: high`; the audited response stopped with `finish_reason: stop`, not `length`, and transport timeout was `180000ms` from adapter params rather than a low default (`configs/cod-test-xiaomi-mimo-v2-omni-openrouter-high-thinking-rerun.yaml`, `server/lib/ai-targets.cjs`, `phase1-gather-context/raw/ai/dialogue-transcription/attempt-01/capture.json`). This does not rule out upstream model behavior, but it argues against our own token/timeout ceiling being the immediate culprit.
- **Audio extraction settings do not show a hidden shortening seam.** Dialogue audio is extracted as MP3 via ffmpeg using `libmp3lame`, `192k`, `44.1kHz`, stereo, and preflight bases chunking on the extracted file’s actual size/duration. The measured audio file is the same `140.042449s` asset the provider received, and the whole-asset lane stayed well under the inline budget (`configs/cod-test-xiaomi-mimo-v2-omni-openrouter-high-thinking-rerun.yaml`, `server/lib/ffmpeg-config.cjs`, `server/lib/audio-preflight.cjs`, `phase1-gather-context/raw/ffmpeg/dialogue/ffprobe-audio-duration.json`, `phase1-gather-context/raw/ffmpeg/dialogue/chunk-plan.json`).

That means the pipeline now exposes the model’s short stop instead of hiding it; it is not the source of the short stop.

Assessment:
- I did **not** find a config switch, audio extraction limit, validator rejection rule, tool-loop cap, or post-processing trim that would plausibly cut a valid 140s whole-asset transcript down to 121s.
- I **did** find two plausible contributors on our side that can allow the upstream mistake to pass through unchallenged:
  1. the whole-asset prompt never anchors the model to the measured runtime, and
  2. the validator contract accepts any structurally valid partial transcript without checking duration/coverage against the known asset length.
- Combined with Task 1’s evidence (`finish_reason: stop`, single whole-asset pass, model reasoning already misestimating duration), the strongest current diagnosis is still **upstream Xiaomi/OpenRouter whole-asset grounding/timing weakness, enabled by a permissive contract rather than caused by a local truncation mechanism**.

**Results:** Synthesized Tasks 1–2 into a ranked diagnosis and recommended the smallest next lane.

Ranked diagnosis:
1. **Upstream Xiaomi grounding/timing weakness** — most likely primary cause. The provider returned a clean `finish_reason: "stop"` on a single whole-asset request, yet stopped coverage at `121s` against a real `140.042449s` file and even reasoned internally that the user had provided a transcript and that the audio was "about 100 seconds long." That is the strongest evidence of model-side grounding/timing weakness rather than local truncation.
2. **Prompt contract weakness** — likely secondary enabler. The whole-asset prompt asks for timestamps and `totalDuration`, but never supplies the real ffprobe-measured runtime or explicitly tells the model to anchor coverage and `totalDuration` to that measured duration. Given the model’s apparent duration guessing, this is the smallest high-leverage seam on our side.
3. **Validator acceptance contract weakness** — likely tertiary enabler. The local validator accepts structurally valid JSON but does not compare `totalDuration`, last segment end, or `coverage.end` against known source duration, so a prematurely ended transcript can pass on the first try.
4. **Token budget** — currently low-probability. The run used `max_tokens: 25000`, generous timeout, and returned `finish_reason: "stop"`, not `"length"`.
5. **Audio extraction settings** — low-probability. The extracted MP3 measured the full `140.042449s`, stayed within whole-asset budget, and no evidence showed truncation during ffmpeg extraction or inline transport prep.
6. **Config / validator loop mechanics** — least likely as root cause. The active config kept dialogue in whole-asset mode with `fallback_to_chunked: false`, and the tool loop succeeded on the first provider turn / first validator call without any local retry exhaustion or loop-induced shortening.

Recommendation: **run the smallest prompt-contract experiment first, not a validator hardening change first.**

Smallest next experiment lane:
- Inject the measured dialogue runtime into the whole-asset prompt (for example: “The attached audio duration is exactly 140.042449 seconds.”).
- Add a direct instruction that `totalDuration` must reflect the full attached audio runtime, not an estimate, and that the transcript should cover the full timeline through the end of the file even when the tail is silent/sparse.
- Re-run the same Xiaomi/OpenRouter whole-asset asset with no other behavioral changes.

Why this should come first:
- It is the **smallest reversible change** touching the seam most plausibly capable of improving the outcome without changing system control flow.
- It directly tests whether Xiaomi can recover when explicitly grounded to the true runtime. Right now the prompt leaves that grounding implicit, and the model has already shown that it guesses duration badly.
- A validator-first change would mostly convert the current behavior into a failure/retry path; it would not tell us whether the model can succeed if better anchored.
- If the runtime-anchored prompt still stops near `121s`, that becomes stronger evidence that the real blocker is upstream model weakness, at which point a validator rejection + fallback/chunking lane becomes justified as the next fix.

Recommended follow-up ordering after that experiment:
- **If anchored rerun succeeds:** keep the prompt improvement and then consider adding validator duration/coverage checks as defense-in-depth.
- **If anchored rerun still fails:** treat upstream Xiaomi whole-asset grounding as confirmed weak for this asset class, then prioritize validator duration-honesty enforcement plus a recovery/fallback lane (likely fail-whole-asset-then-chunk rather than silently accepting a short transcript).
